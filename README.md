2. **Подробное описание классов**

2.1. *Класс `Content` (Абстрактный базовый класс)*

Атрибуты:
- `_id`: Уникальный идентификатор контента (UUID) — Принцип ООП: Инкапсуляция
- `_text`: Текстовое содержимое — Принцип ООП: Инкапсуляция
- `_author_id`: ID автора контента — Принцип ООП: Инкапсуляция
- `_timestamp`: Время создания (формат YYYY-MM-DD HH:MM:SS) — Принцип ООП: Инкапсуляция

Методы:
- `render()`: Возвращает отформатированное представление — Принцип ООП: Полиморфизм
- `to_dict()`: Сериализация в словарь для JSON — Принцип ООП: Полиморфизм
- `from_dict()`: Десериализация из словаря — Принцип ООП: Полиморфизм

Назначение: Базовый класс для всех типов контента. Не может быть создан напрямую (абстрактный).

2.2. *Класс `Post` (наследник Content)*

Атрибут: `_likes`: Количество лайков на публикации — Принцип ООП: Инкапсуляция

Методы:
- `like()`: Увеличивает счётчик лайков на 1
- `render()`: Переопределённый метод отображения — Принцип ООП: Полиморфизм
- `to_dict()`: Сериализация с полем "type": "Post" — Принцип ООП: Полиморфизм

Назначение: Представление публикации в ленте новостей.

2.3. *Класс `Message` (наследник Content)*

Атрибуты:
- `_receiver_id`: ID получателя сообщения — Принцип ООП: Инкапсуляция
- `_is_read`: Статус прочтения (по умолчанию False) — Принцип ООП: Инкапсуляция

Методы:
- `mark_as_read()`: Отмечает сообщение как прочитанное
- `render()`: Переопределённый метод со статусом — Принцип ООП: Полиморфизм
- `to_dict()`: Сериализация с полем `"type": "Message"` — Принцип ООП: Полиморфизм

Назначение: Представление личного сообщения между пользователями.

2.4. *Класс `User`*

Атрибуты:
- `__password` (Атрибут): Пароль (приватный, двойное подчёркивание) — Принцип ООП: Инкапсуляция
- `_friends_ids` (Атрибут): Список ID друзей — Принцип ООП: Инкапсуляция
- `_posts` (Атрибут): Список публикаций пользователя — Принцип ООП: Агрегация
- `_inbox` / `_outbox` (Атрибут): Входящие/исходящие сообщения — Принцип ООП: Агрегация

Методы:
- `check_password()`: Проверка пароля без возврата значения — Принцип ООП: Инкапсуляция
- `change_password()`: Смена пароля с проверкой старого
- `add_friend()`: Добавление друга с проверкой дубликатов
- `remove_friend()`: Удаление друга из списка
- `get_friends_ids()`: Возвращает копию списка (защита)
- `create_post()`: Создание и добавление публикации
- `send_message()`: Отправка сообщения (сохранение в outbox)
- `receive_message()`: Получение сообщения (добавление в inbox)
- `get_inbox()`: Возврат входящих с mark_as_read()
- `get_outbox()`: Возврат исходящих сообщений
- `export_profile_to_file()`: Экспорт в TXT файл — Работа с файлами
- `to_dict()` / `from_dict()`: Сериализация/десериализация

Назначение: Модель пользователя со всеми связанными данными.

2.5. *Класс `SocialNetwork`*

Атрибуты:
- `_users`: Словарь всех пользователей (ключ: user_id) — Принцип ООП: Инкапсуляция
- `_current_user`: Текущий авторизованный пользователь — Принцип ООП: Инкапсуляция
- `DB_FILE`: Имя файла базы данных (`network_db.json`)

Методы:
- `__init__()`: Инициализация с загрузкой БД
- `_load_db()`: Загрузка из JSON при старте — Работа с файлами
- `_save_db()`: Сохранение в JSON при изменениях — Работа с файлами
- `register()`: Регистрация нового пользователя
- `login()`: Авторизация по логину и паролю
- `logout()`: Сброс текущего пользователя
- `find_user_by_username()`: Поиск по логину
- `find_user_by_id()`: Поиск по UUID
- `get_current_user()`: Получение текущей сессии

Назначение: Класс-менеджер (Controller), управление всей сетью.

3. **Диаграмма взаимодействий**

*Основной поток взаимодействия (Linear Flow)*:

`Console Interface` (Input/Output) → `SocialNetwork` (Controller/Logic) → `User` (Model/Data) → `Content Objects` (Post/Message) → `JSON File` (Persistence)

*Детализация потоков данных*:
- Ввод данных: Пользователь → Console Interface → SocialNetwork
- Обработка логики: SocialNetwork → User (поиск, авторизация, методы)
- Работа с контентом: User → Content (создание Post/Message, рендеринг)
- Сохранение состояния: SocialNetwork → JSON File (сериализация _users)
- Вывод результата: SocialNetwork → Console Interface → Пользователь

*Цикл жизни запроса*:
- `Input` (Консоль)
- `Process` (Контроллер SocialNetwork)
- `Action` (Модель User/Content)
- `Store` (Файловая система)
- `Output` (Консоль)

# Краткое содержание

## Суть задания
Необходимо написать программу, моделирующую работу социальной сети, которая выполняет следующие действия:
1. Реализует архитектуру на основе принципов ООП (инкапсуляция, наследование, полиморфизм)
2. Обеспечивает регистрацию пользователей, управление друзьями, публикацию постов и обмен сообщениями
3. Организует постоянное хранение данных в файловом формате и экспорт профиля.
Предоставляет пользователю консольный интерфейс для взаимодействия с системой

## Описание входных данных
**Особенность данных:** программа предназначена для интерактивной работы с пользователем через консоль и постоянного хранения состояния системы. Для корректной обработки текстов на русском языке (имена, сообщения, посты) используется кодировка UTF-8. Основным хранилищем служит файл `network_db.json`, который накапливает данные о пользователях и контенте. В процессе работы программа также генерирует уникальные идентификаторы (UUID) для объектов и временные метки для событий.

## Алгоритм решения
Программа реализует следующий алгоритм:

**Шаг 1: Построение иерархии классов**

Создаётся абстрактный базовый класс `Content`, определяющий общую структуру контента. От него наследуются классы `Post` (публикация) и `Message` (сообщение). Это реализует принцип наследования и полиморфизма через методы `render()` и `to_dict()`, которые работают по-разному для каждого типа контента.

**Шаг 2: Инкапсуляция данных пользователя**

Класс `User` скрывает чувствительные данные (пароль) через приватные атрибуты (например, `__password`). Доступ к функционалу (добавление друзей, создание постов) осуществляется только через публичные методы, что защищает внутреннее состояние объекта от некорректного изменения извне.

**Шаг 3: Управление состоянием сети**

Класс `SocialNetwork` выступает в роли контроллера. При инициализации он загружает данные из файла `network_db.json` (десериализация). При любом изменении данных (регистрация, пост, сообщение) вызывается метод `_save_db()`, который преобразует объекты в словари и записывает их обратно в JSON-файл (сериализация).

**Шаг 4: Консольный интерфейс и навигация**

Блок `if __name__ == "__main__":` запускает цикл меню. Функции `menu_auth` и `menu_user` организуют навигацию. Программа ожидает ввод цифр и текста от пользователя, валидирует данные и вызывает соответствующие методы классов сети для выполнения действий.

## Ожидаемые результаты
Программа выводит результат в двух форматах:
1. **Файловый:**
в директории проекта создаётся и обновляется файл `network_db.json`, содержащий полную копию базы данных (пользователи, связи, контент) в формате JSON.
По запросу пользователя создаётся файл `username_profile.txt`, содержащий текстовый отчёт о профиле, последних публикациях и сообщениях.
2. **Консольный:**
выводится интерактивное меню с пунктами навигации.
После выполнения действий (например, «Друг добавлен», «Сообщение отправлено») выводятся подтверждающие сообщения.
При просмотре ленты или сообщений отображается отформатированный текст контента с указанием времени и автора.
